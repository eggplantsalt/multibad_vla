# 全局协作原则

你正在一个已有代码库上做“最小侵入式增量开发”，不是重写项目。

必须遵守以下规则：

1. 先读 `docs/CONTEXT.md` 和 `docs/OVERVIEW.md`，再开始工作。
2. 长任务采用“先探索、再计划、再实现”的工作流。第一次只允许阅读代码和写计划，不允许直接改代码。
3. 每次只实现一个模块，禁止一次性跨多个模块大改。
4. 不要删除或破坏现有训练与评测流程。所有新功能必须默认关闭，只有显式传参时才启用。
5. 不要改 TFDS/RLDS 底层数据管线，除非我明确要求。优先在 PyTorch 侧做 wrapper、collate、训练 loop 集成。
6. 不要做大规模重构，不要引入与任务无关的抽象，不要额外“优化代码风格”。
7. 任何新增功能都必须给出：
   - 修改文件列表
   - 关键类/函数说明
   - 可运行命令
   - 最小验证步骤
   - 写入 `docs/PROGRESS.md`
8. 若需要做架构选择，先把候选方案写入 `docs/PLAN.md`，等我确认后再实现。
9. 不要做破坏性操作，不要删除文件，不要重命名大目录，不要改已有 checkpoint 兼容逻辑。
10. 每次完成后，都要更新：
   - `docs/PLAN.md`
   - `docs/PROGRESS.md`
   - `docs/DECISIONS.md`
   - `docs/VALIDATION.md`



# 运行与验证约束

注意：你当前工作的环境不能连接远程训练服务器，因此你不要执行任何训练、评测、dry-run、smoke test 或伪造验证结果。

你的职责是：
1. 阅读代码并保证接口、参数、导入、日志字段和保存逻辑自洽；
2. 完整实现本轮要求的代码；
3. 提供我在远程服务器上运行所需的命令；
4. 明确说明预期日志、预期输出文件、以及若失败应优先检查的模块。

禁止：
- 声称你已经运行了代码
- 声称某命令“通过了”，除非你只是静态检查且明确说明
- 写虚假的示例运行结果
- 增加与当前轮次无关的功能
